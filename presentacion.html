01 — Introducción

¿Qué es Ecommify?

🛒

Plataforma e-commerce multi-vendor

Conecta compradores y vendedores de productos tecnológicos. Los vendedores gestionan su inventario, los compradores exploran catálogos, agregan al carrito y realizan pedidos.

🎓

Contexto académico

Proyecto de Fundamentos de Testing en la Maestría en Arquitectura de Software. Dataset real: Brazilian E-Commerce (Olist) de Kaggle con datos de transacciones reales.

🗄️

Arquitectura Polyglot Persistence

Dos bases de datos especializadas: PostgreSQL para transacciones ACID y MongoDB para catálogo, reseñas y comportamiento de usuario.

🧪

Estrategia de pruebas integral

67 pruebas unitarias con TDD + 6 escenarios de rendimiento con JMeter. Todo integrado en un pipeline CI/CD con GitHub Actions.

Node.js 20 PostgreSQL 16 MongoDB 7 React 18 Docker Compose JMeter 5.4.3 Jest 29.7

04b — Pruebas unitarias en detalle

Anatomy of a test — patrón BDD + AAA

// auth.service.test.js — Given/When/Then

describe('AuthService', () => {

  describe('validateEmail', () => {

    test('Given valid email,
      When validateEmail is called,
      Then returns true', () => {

      // Arrange
      const email = 'user@example.com';

      // Act
      const result = validateEmail(email);

      // Assert
      expect(result).toBe(true);
    });

  });
});

¿Por qué mocking total?

  • Los tests corren en <20 segundos sin levantar bases de datos
  • Cada test es determinístico — mismo input, mismo output siempre
  • Permite correr en CI/CD sin infraestructura costosa
  • Fuerza a inyectar dependencias — mejor diseño de código

Estructura del test en jest.config.js

coverageThreshold: {
  global: {
    branches:   75,
    functions:  75,
    lines:      75,
    statements: 75
  }
}
Umbrales mínimos del 75% — el equipo superó 98.3%

05c — Hallazgos de rendimiento

Hallazgos y cuellos de botella

✅

Fortalezas encontradas

  • Sistema sólido hasta 100 VUs — todos los SLOs cumplidos
  • Soak de 60 minutos: sin memory leaks, p95 estable en 26ms
  • Recuperación tras spike de 300 VUs en menos de 2 minutos
  • Prueba de regresión 60% más rápida que el baseline (25ms vs 62ms)
⚠️

Cuellos de botella

  • PostgreSQL connection pool — se satura a 400 VUs, timeouts de 10s
  • Node.js event loop — single-thread con alta I/O genera latencia lineal +200 VUs
  • Windows TCP limit — puertos efímeros 49152-65535 = solo 16K conexiones disponibles
🐛

Defectos registrados (PERF-01 a 05)

💡

Recomendaciones de mejora

  • Aumentar pool.max de PostgreSQL a 50-100 conexiones
  • Implementar clustering Node.js o usar PM2 para multi-core
  • Agregar Redis como cache L2 para catálogo de productos
  • Configurar TCP ephemeral port range en Windows o usar Linux para tests
05d — Pipeline CI/CD

Pipeline de integración continua

name: CI Pipeline - Ecommify

on:
  push:
    branches: ["main", "master"]
  pull_request:
    branches: ["main", "master"]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'

      - run: npm install
      - run: npm test -- --coverage --forceExit

      - uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          retention-days: 14

Flujo del pipeline

🔀 Push / Pull Request a main
Disparador automático en cada cambio
📦 Checkout + Node.js 18
Instala dependencias desde package-lock.json con caché
🧪 67 tests + cobertura
npm test --coverage --forceExit · ~2-3 minutos
🚫 Falla si coverage < 75%
Thresholds en jest.config.js bloquean el merge
📤 Artifact: reporte HTML
Coverage report disponible 14 días en GitHub Actions
06 — Conclusiones

Aprendizajes del proyecto

Sobre pruebas unitarias

  • TDD desde el inicio fuerza a diseñar código modular y testeable — los módulos bien separados alcanzaron 100% de cobertura con naturalidad
  • Mocking total permite 67 tests en 20s sin levantar infraestructura — crítico para CI/CD ágil
  • La matriz de trazabilidad conecta cada fallo con su historia de usuario, reduciendo el tiempo de diagnóstico
  • El patrón BDD (Given/When/Then) hace los tests legibles para stakeholders no técnicos

Sobre pruebas de rendimiento

  • El baseline es la prueba más importante — sin él no hay referencia para medir degradación
  • Definir SLOs antes de probar, no después — definirlos a posteriori sesga el criterio de aceptación
  • El soak test (60 min) reveló lo que ninguna otra prueba mostraría: ausencia de memory leaks
  • Los cuellos de botella no siempre son el código — el connection pool y el SO fueron los límites reales
"Ningún tipo de prueba es suficiente por sí solo. Las pruebas unitarias validan la corrección del código; las pruebas de rendimiento validan la capacidad operacional. Juntas, dan confianza real para llevar el sistema a producción."
